## What Went Well

1. **Optimistic mutation pattern established and reused.** The `onMutate / onError / onSettled` template — snapshot, optimistic update, rollback on failure, invalidate on settled — was established in `TodoItem` (toggle + delete) and reused identically in `TodoInput` (create). Epic 3 will use the same shape.

2. **CSS custom property discipline.** All 11 color tokens defined in Story 2.1 were used without exception throughout the epic. No hardcoded hex values appear in any component. Visual consistency is solid.

3. **`React.forwardRef` on UI primitives paid off.** Establishing `forwardRef` on `Input` in Story 2.1 felt premature, but it was exactly the mechanism needed in Story 2.5 to forward a ref from `TodoInput` for post-submit focus restoration.

4. **Two clean test strategies emerged and held.** `vi.mock('../hooks/useTodos')` for state-control-only tests (`TodoList`); real `QueryClientProvider` + `vi.stubGlobal('fetch', vi.fn())` for mutation-path tests (`TodoItem`, `TodoInput`). No test file deviated from one of these two patterns.

5. **Accessibility-first from the start.** `aria-busy`, `aria-live="polite"`, `role="list"`, `role="listitem"`, labelled checkboxes and delete buttons — all added in the delivery stories, not retrofitted.

6. **Tailwind 4 CSS-first worked cleanly after initial orientation.** Once the v4 break (no `tailwind.config.js`, `@import "tailwindcss"`, `[property:var(--token)]` syntax) was understood in Story 2.1, subsequent stories had no CSS configuration issues.

7. **Incremental story structure was clean.** `TodoList` rendering a placeholder `<div>` while `TodoItem` was deferred (Story 2.3) kept each story focused. Story 2.4 replaced it without friction.

---

## Challenges

1. **Tailwind 4 orientation cost.** The v4 CSS-first config is a hard break from v3. It required explicit documentation in Story 2.1's dev notes and careful re-reading in every story that extended the CSS. Future devs joining the project need to be briefed on the v4 pattern.

2. **Compound Input border override is a leaky abstraction.** `Input.tsx` applies `border`, `background`, and `outline` via inline styles. When nested inside `TodoInput`'s compound container (which provides its own `focus-within` ring), those inner styles needed explicit zeroing. It works, but the pattern is fragile if `Input` is refactored.

3. **ARIA label discrepancy to resolve.** Story 2.5 implemented `aria-label="Add task form"` on the `<form>` element. `epics.md` specifies `aria-label="Add a task"`. The test matches the story file. This needs explicit resolution before Epic 4's ARIA audit.

4. **`TodoList` error state has no retry mechanism.** Currently renders a hardcoded `"Something went wrong. Please try again."` paragraph with no retry action. This is an intentional Epic 3 dependency, but the current state is user-hostile for any API failure. Epic 3 Story 3.4 addresses this directly.

## Technical Patterns Established

These patterns are now project standards. All future stories should follow them.

| Pattern | Where Defined | Notes |
|---|---|---|
| `onMutate / onError / onSettled` optimistic mutation | `TodoItem.tsx`, `TodoInput.tsx` | Snapshot cache before mutation, restore on error, invalidate on settled |
| `vi.mock` hook for state-driven component tests | `TodoList.test.tsx` | Use when component only reads from a hook |
| Real `QueryClientProvider` + mocked `fetch` for mutation tests | `TodoItem.test.tsx`, `TodoInput.test.tsx` | Use when component calls a mutation |
| `renderWithQuery` helper | All component tests | Wraps `QueryClientProvider` with test-safe config |
| `React.forwardRef` + `...props` spread on all UI primitives | `ui/Button.tsx`, `ui/Input.tsx`, `ui/Checkbox.tsx` | Enables parent-controlled refs and className overrides |
| CSS custom properties only | `globals.css`, all components | `var(--color-*)`, `var(--font-*)`, `var(--spacing-*)` |
| `ApiError` class with `.status` | `api/types.ts` | Typed error with HTTP status for Epic 3 error message routing |

## Epic 3 Preparation

**Epic 3: Error Handling & Resilience** — 4 stories.

### Dependency Map

Story 3.1 (`InlineError` component) has no dependencies and must be completed before Stories 3.2–3.4, which all wire `InlineError` into existing code.

```
3.1 InlineError component (standalone)
  └── 3.2 Create failure UX (wires into TodoInput.onError)
  └── 3.3 Toggle/delete failure UX (wires into TodoItem.onError)
  └── 3.4 List load failure + retry (replaces TodoList error paragraph)
```

### What Epic 2 Prepared

- `TodoInput.tsx` `onError` → currently does cache rollback only; Story 3.2 adds `InlineError` state here. Note: `setText('')` fires only in `onSuccess`, so **the user's typed text is preserved on failure**. Story 3.2 shows this preserved text alongside `InlineError` — no restoration needed.
- `TodoItem.tsx` `onError` (toggle + delete) → currently does cache rollback only; Story 3.3 adds `InlineError` state here.
- `TodoList.tsx` `isError` branch → currently renders a plain paragraph; Story 3.4 replaces it with `InlineError` + retry button.
- `ApiError.status` + `throwIfNotOk` → `error.message` already contains the server error text; Stories 3.x can use it directly for `InlineError` copy.

### Epic 3 Testing Pattern Extension

Epic 3 tests will extend the existing patterns with rejection mocks:
- Network failure: `vi.fn().mockRejectedValueOnce(new Error('Network error'))`
- Server error: `vi.fn().mockResolvedValueOnce({ ok: false, status: 500, json: () => Promise.resolve({ error: '...' }) })`

No new packages required.
