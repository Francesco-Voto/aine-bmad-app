## What Went Well

1. **Story sequencing was clean.** 5.1 → 5.2 → 5.3 → 5.4 built on each other without friction. Each story's "What Already Works" table was accurate — no story had to revisit a prior story's outputs.

2. **Named volume decision in 5.3 avoided a class of problems.** The story originally considered a bind mount (`./data:/app/data`). Switching to a named Docker volume (`todo-data:/app/data`) preemptively solved the non-root user write-permission problem on the host directory — no `chmod`/`chown` workaround needed at runtime.

3. **`docker-compose.override.yml` auto-merge pattern was ergonomic.** Dev environment configuration (`NODE_ENV=development`, `DB_PATH=/app/data/dev.db`) just works by default on `docker-compose up`. The test stack requires an explicit `-f` flag — correct separation of defaults from special-purpose configs.

4. **Build context architecture handled the monorepo correctly.** Both Dockerfiles use the monorepo root as build context, copying only the workspace manifests needed for `npm ci -w <workspace>`. Layer caching is correct: package manifests land in an early layer, source files in a late layer.

5. **`better-sqlite3` native addon handled cleanly.** Stage 2 runs `npm ci --omit=dev` fresh (not copying `node_modules` from Stage 1) and installs `python3 make g++` only for the duration of compilation, then removes them. Final production image has no build toolchain.

6. **E2E Playwright config split was the right call.** Dev config uses `webServer` to spin up the stack. Docker config (`playwright.docker.config.ts`) assumes the stack is already up and sets `baseURL: http://localhost`. The existing E2E specs required zero changes — only the config differs.

## Challenges & Implementation Deviations from Story Templates

1. **Monorepo build context.** Story Task 1 templates were written assuming `docker build ./server` (workspace as context). The actual build requires the monorepo root as context to access `package-lock.json` and workspace manifests. Both Dockerfiles use `docker build -f <workspace>/Dockerfile .` from the root. The story templates were updated mid-sprint to reflect this.

2. **Nginx DNS resolution for Docker service discovery.** The story template used `proxy_pass http://server:3000/api/` directly. In practice, if Nginx resolves `server` at startup before the `server` container is resolvable, the proxy silently fails. The actual `nginx.conf` uses Docker's embedded DNS resolver (`resolver 127.0.0.11 valid=30s`) and defers resolution to request time via `set $backend`. This is a Docker networking subtlety not capturable in a spec.

3. **Nginx `USER` directive.** The story specified `USER nginx` in the `client/Dockerfile`. Nginx's master process must bind port 80 as root; worker processes already drop to the `nginx` user via the global `nginx.conf` `user nginx;` directive. Adding an explicit `USER nginx` before `CMD` causes a bind-permission error. The final Dockerfile omits the directive and documents why.

4. **`npm run docker:up` from workspace directories.** The Docker scripts only existed in the root `package.json`, so running them from `server/` or `client/` failed. Fixed during the retrospective by adding delegate scripts (`npm run docker:* --prefix ..`) to both workspace `package.json` files.

## Technical Patterns Established

1. **Monorepo Docker build context pattern.** Build context = monorepo root; workspace-scoped `npm ci -w <workspace>` for dependency isolation; source files copied after manifests for correct layer caching.

2. **Native addon container pattern.** Install build tools → `npm ci --omit=dev` → remove build tools in the same `RUN` chain or a subsequent layer before `USER` switch. Do not copy `node_modules` across stages.

3. **Nginx deferred DNS resolution.** Always use `resolver 127.0.0.11 valid=30s` + `set $backend` for Docker service-name proxying. Direct `proxy_pass http://service:port` is fragile at container startup.

4. **Named volumes over bind mounts for non-root containers.** Named volumes are owned by the Docker daemon and writable by the container user. Bind mounts inherit host ownership and require explicit `chown` or UID matching.

5. **Workspace-delegating npm scripts.** Docker and other root-level scripts added to workspace `package.json` using `--prefix ..` so developers working in any directory have a consistent interface.
